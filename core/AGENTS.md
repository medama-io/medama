# AGENTS.md

## Role

You are working on `core`, the Go backend API server and deployable single binary for Medama. Preserve the API, database, migration, and runtime configuration contracts unless the user explicitly asks to change them.

## Success Criteria

- The Go module still builds and tests with the repository tool versions.
- API behavior remains aligned with `openapi.yaml`.
- SQLite and DuckDB behavior remains compatible with existing data and migrations.
- Any generated Go code is regenerated from source inputs, not edited directly.
- The final change includes targeted validation for the touched package or a clear reason validation could not run.

## Package Boundaries

- `cmd`: CLI flags, environment config, startup, shutdown, and server wiring.
- `services`: API handler behavior and coordination across auth, databases, parsing libraries, runtime settings, and assets.
- `db/sqlite`: application metadata persistence.
- `db/duckdb`: analytics/event persistence and stats queries.
- `migrations`: SQLite and DuckDB schema evolution.
- `middlewares`: HTTP middleware for auth, logging, compression, CORS, recovery, cache/version headers, and rate limiting.
- `model`: shared domain types and error values.

## Constraints

- Run Go commands from `core` unless using root monorepo tasks such as `mise run core:test`.
- CGO is required for the database stack.
- Keep control flow direct: handle errors early, keep the normal path minimally indented, and return useful error context.
- Pass `context.Context` explicitly for request-scoped work. Do not store contexts in structs.
- Prefer concrete types and small helpers. Add interfaces only at the consumer side when they solve a real testing or ownership problem.
- Avoid new dependencies, reflection, clever generics, or goroutines unless the local package already uses that pattern and the task needs it.

## Generated Code and Assets

- `openapi.yaml` is the API contract.
- `api/oas_*_gen.go` is generated by ogen. Do not edit it by hand.
- If `openapi.yaml`, `.ogen.yml`, `generate.go`, `generate.sh`, `go.mod`, or `go.sum` changes, run `mise run generate:go`.
- Run `mise run generate` when dashboard or tracker embedded assets also need refreshing.
- Keep the `generate.sh` Beacon API `text/plain` compatibility patch in the generation workflow.

## Validation Defaults

- Prefer targeted `go test` commands for changed packages.
- Prefer existing in-package database helpers, fixtures, and snapshots over ad hoc mocks or setup.
- Use `mise run test` for broad backend changes; it runs Go generation first.
- Use `mise run lint:fix`, then `mise run lint`, for lint cleanup.
- If migrations change, add or update migration coverage and keep `migrations/Schema.md` consistent.
- If dependencies change, run `go mod tidy`.
- DuckDB stats tests use snapshots. Update snapshots only for intentional output changes.

## Stop Rules

- Stop before changing API shapes, auth/session semantics, config flags, migration ordering, database storage format, or generated handlers unless the task explicitly calls for it.
- Stop if the next step needs a production service, unavailable secret, or destructive local data operation.
